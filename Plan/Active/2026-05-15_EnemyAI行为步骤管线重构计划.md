# 2026-05-15 EnemyAI 行为步骤管线重构计划

## 背景

当前 `EnemyAutoAISystem` 已经承担了完整敌人决策：

- 选择仇恨目标。
- 构建可破墙 / 不可破墙距离场。
- 判断敌人类型。
- 验证攻击范围。
- 构造攻击和移动 Intent。
- 处理诅咒锁核心箱。
- 处理二筒半血分裂。

这套实现适合早期 demo，但新增敌人时会继续把类型判断塞进同一个系统。更重要的是，如果后续要演进到选项式状态机或行为树，不能只按“每种敌人一个策略类”拆。那会把行为继续封在黑盒里，后续无法自然做可视化编排。

本次重构目标不是立即实现行为树，而是先把敌人 AI 拆成可组合的行为步骤。行为步骤是 A 层原子，后续可以被 C 层选项式状态机和 B 层行为树复用。

```text
B 行为树 / 可视化编排
  uses
C 选项式状态机 / 优先级决策
  uses
A 行为步骤 / 条件 / 查询 / 行动原子
  uses
ECS / Intent / DistanceField / EntitySystem
```

## 设计原则

1. 行为步骤优先按“决策管线阶段”拆，不按敌人种类拆。
2. 条件验证必须尽量无副作用，未来可以直接成为行为树 Condition。
3. 查询步骤只生产候选，不提交 Intent。
4. 行动步骤只构造决策结果，不直接写入 `IntentSystem`。
5. 提交 Intent 只保留在统一提交层，便于节拍、预览、回放和调试。
6. 不绕过现有 Intent / Resolution 约定。移动、攻击、刷怪仍走 Intent 执行。
7. 特殊行为如果会改世界，优先改成 Intent 或 deferred effect；短期保留时必须标记技术债。

## 目标形态

`EnemyAutoAISystem` 最终降级为调度器：

```text
EnemyAutoAISystem.Tick
  -> BuildWorldFacts
  -> BuildDistanceFields
  -> RefreshGlobalEnemyState
  -> foreach enemy
       -> BuildEnemyDecisionContext
       -> ResolveEnemyProfile
       -> RunDecisionPipeline
       -> CommitDecision
```

敌人差异通过 profile / pipeline 表达：

```text
Go:
  TryAttackBlockingPlayer(Adjacent)
  TryAttackAggroTarget(Adjacent)
  TryChaseOrBreakWall(move=1, canBreakWall=true)

Guokui:
  TryAttackBlockingPlayer(Adjacent)
  TryAttackAggroTarget(Adjacent)
  TryChaseOrBreakWall(move=2, canBreakWall=true)

Grenadier / Crossbow:
  TryAttackBlockingPlayer(LineRequiresWall)
  TryAttackAggroTarget(LineRequiresWall)
  TryMoveToFutureAttackPoint(LineRequiresWall)
  TryChaseOrBreakWall(move=1, canBreakWall=false)

Artillery:
  TryAttackAggroTarget(LineRequiresWall + Square2x2)
  TryMoveToFutureAttackPoint(LineRequiresWall)
  TryChaseOrBreakWall(move=1, canBreakWall=false)

CurseCaster:
  TryMaintainCurseLock
  Noop

Ertong:
  TrySplitOnLowHealth
  TryAttackBlockingPlayer(Adjacent)
  TryAttackAggroTarget(Adjacent)
  TryChaseOrBreakWall(move=4, canBreakWall=true)
```

## 行为步骤分层

### 1. Facts：事实生产

只负责读取世界并整理上下文，不做行为决策。

候选类型：

- `EnemyDecisionWorld`
- `EnemyDecisionContext`
- `EnemyDecisionBlackboard`
- `EnemyDistanceFieldSet`
- `EnemyProfile`

当前来源：

- `TryFindAggroTarget`
- `EnsureDistanceFields`
- `BuildDistanceField`
- `RefreshCurseLocks`
- 当前敌人的 `CoreComponent`
- 当前敌人的 `PropertyComponent`
- 当前敌人的 `StatusComponent`
- `_reservedMoveCells`

### 2. Conditions：条件验证

纯验证节点，原则上不分配 Intent、不改世界。

第一批条件：

- `IsAdjacentToTarget`
- `IsInLineRange`
- `IsInDiagonalRange`
- `HasWallBetween`
- `CanBreakWalls`
- `CanUseRangedAttack`
- `CanUseArtilleryAttack`
- `IsHealthBelowRatio`
- `IsCellEnterable`
- `IsCellReserved`
- `IsValidAttackTarget`

当前需要拆开的混合函数：

- `TryBuildAttackAgainstTarget` 同时做了范围验证和 Intent 构造。
- `TryMoveTowardFutureAttackPoint` 同时做了站位搜索和 MoveIntent 构造。
- `TryGetNextAction` 同时做了追击、破墙、攻击占位者和移动距离选择。

### 3. Queries：候选生成

只回答“可以打谁 / 可以去哪 / 可以怎么走”，不提交行动。

第一批查询：

- `SelectAggroTarget`
- `SelectBlockingPlayerTarget`
- `FindNearestAttackStandPoint`
- `SelectStepTowardPoint`
- `SelectDistanceFieldAction`
- `SelectMoveDistance`
- `SelectSplitSpawnCell`

建议输出结构：

```text
EnemyAttackCandidate
  target cells
  attack pattern
  score

EnemyMoveCandidate
  direction
  distance
  destination
  score

EnemySpecialCandidate
  kind
  payload
  consumesTurn
```

### 4. Actions：决策构造

行动步骤把候选转换成 `EnemyDecision`，但不直接调用 `IntentSystem.SetIntent`。

第一批行动：

- `BuildAdjacentAttackDecision`
- `BuildLineAttackDecision`
- `BuildArtillerySquareAttackDecision`
- `BuildMoveDecision`
- `BuildWallAttackDecision`
- `BuildNoopDecision`
- `BuildMaintainCurseLockDecision`
- `BuildSplitDecision`

建议统一输出：

```text
EnemyDecision
  IntentType
  IntentPayload
  ReservedDestination
  HasReservedDestination
  ConsumesTurn
  DebugLabel
```

`IntentPayload` 第一版可以直接持有已申请的 `Intent`，但更推荐持有轻量 payload，由提交层统一申请池对象：

```text
MovePayload(direction, distance)
AttackPayload(targets, multipliers)
SpawnPayload(origin, bp)
```

这样行为步骤不会依赖 `IntentSystem.Request<T>()`，未来可视化和 dry-run 更容易。

### 5. Composites：组合步骤

组合步骤是 C 层状态机 / B 层行为树的过渡层。第一版可以用代码组合，后续再资产化。

第一批组合：

- `TrySpecialBeforeNormal`
- `TryAttackBlockingPlayer`
- `TryAttackAggroTarget`
- `TryMoveToFutureAttackPoint`
- `TryChaseOrBreakWall`
- `TryFallbackNoop`

组合结构示例：

```text
Selector
  Sequence TrySpecialBeforeNormal
  Sequence TryAttackBlockingPlayer
  Sequence TryAttackAggroTarget
  Sequence TryMoveToFutureAttackPoint
  Sequence TryChaseOrBreakWall
  Sequence TryFallbackNoop
```

## 分阶段实施

### Phase 0：加保护测试 / 基准记录

目标：重构前先固定当前行为，不用一次补完整测试，但要有可比对基准。

任务：

1. 记录现有敌人行为样例：
   - 近战追核心箱。
   - 近战攻击挡路玩家。
   - 近战破墙。
   - Crossbow / Grenadier 隔墙直线攻击。
   - Artillery 2x2 攻击。
   - CurseCaster 斜向锁核心箱。
   - Ertong 半血分裂。
2. 准备至少一个可重复运行的场景或轻量验证入口。
3. 标记当前 Ertong 直接改世界为技术债。

验收：

- 重构后上述样例行为不发生非预期变化。
- 能快速定位是距离场、攻击验证、移动决断还是特殊行为变化。

### Phase 1：抽 Context 和 Profile，不改行为

目标：先把隐式字段变成显式输入。

任务：

1. 新增 `EnemyDecisionContext`。
2. 新增 `EnemyProfile`，先从现有 tag / BP 名字推导：
   - `MoveDistance`
   - `CanBreakWalls`
   - `AttackPattern`
   - `AttackRange`
   - `RequiresWallBetween`
   - `SpecialKind`
3. 保留现有 `IsEnemyKind`，但只允许在 profile resolver 中出现。
4. `WriteEnemyIntents` 改为读取 profile，不再在主流程到处散落 `isGrenadier` / `isCrossbow` 等局部判断。

验收：

- `EnemyAutoAISystem` 主循环里不再直接判断具体敌人名。
- 行为结果和当前版本一致。

### Phase 2：拆攻击验证和攻击构造

目标：把“验证攻击范围”和“构造攻击 Intent”拆开。

任务：

1. 拆出 line range / adjacent / diagonal / wall-between 条件。
2. 拆出攻击 pattern：
   - `AdjacentSingleCell`
   - `LineSingleCellRequiresWall`
   - `LineSquare2x2RequiresWall`
3. `TryBuildAttackAgainstTarget` 改为：
   - profile 选择 pattern。
   - pattern 验证。
   - pattern 输出 `AttackPayload`。
4. 攻击行为不直接申请 `AttackIntent`，由提交层申请。

验收：

- Crossbow / Grenadier / Artillery 共享 line 条件。
- Artillery 只特殊在 payload shape，不复制整套远程逻辑。

### Phase 3：拆移动候选和移动构造

目标：移动步骤可以被远程站位、追击、破墙复用。

任务：

1. 拆出 `FindNearestAttackStandPoint` 查询。
2. 拆出 `SelectStepTowardPoint` 查询。
3. 拆出 `SelectDistanceFieldAction` 查询，输出：
   - move
   - wall attack
   - occupant attack
4. 拆出 `BuildMoveDecision` / `BuildWallAttackDecision`。
5. 统一处理 reserved destination。

验收：

- 远程单位移动到未来攻击位不再直接构造 `MoveIntent`。
- 近战追击 / 破墙 / 攻击占位者走同一候选结构。
- `_reservedMoveCells` 只在提交层写入。

### Phase 4：特殊行为步骤化

目标：诅咒和分裂变成可组合步骤。

任务：

1. `MaintainCurseLock` 拆成：
   - condition：是否核心箱、是否斜向 3 格、是否已锁。
   - action：记录锁定并消耗行动。
2. `RefreshCurseLocks` 保留为全局事实刷新。
3. `ErtongSplit` 拆成：
   - condition：血量低于 50%。
   - query：寻找分裂格。
   - action：生成 split decision。
4. 长期方案：新增 `TransformIntent` 或 `SplitIntent`，避免 AI Tick 直接 `DestroyEntity` / `CreateEntity`。

验收：

- CurseCaster 不再散落在攻击 / 移动函数里特殊排除。
- Ertong 分裂行为有明确技术债边界，后续可以替换成 Intent。

### Phase 5：管线配置化

目标：敌人类型声明步骤组合，而不是写完整策略类。

任务：

1. 新增 `EnemyDecisionPipeline`。
2. 新增代码版 pipeline registry：
   - `GoPipeline`
   - `GuokuiPipeline`
   - `LineRangedPipeline`
   - `ArtilleryPipeline`
   - `CurseCasterPipeline`
   - `ErtongPipeline`
3. registry 只负责 profile -> pipeline。
4. 资产化暂缓，不引入编辑器可视化。

验收：

- 新增一个敌人时，优先新增或复用步骤组合。
- 如果只改移动距离 / 攻击模式，不需要改 `EnemyAutoAISystem` 主流程。

### Phase 6：为 C/B 留接口

目标：不实现行为树，但接口形状不能堵死行为树。

任务：

1. 让步骤接口接近行为树叶子节点：

```text
Evaluate(context, blackboard) -> StepResult
```

2. `StepResult` 至少支持：
   - `Success`
   - `Failure`
   - `DecisionReady`
3. 组合节点先代码实现：
   - Selector
   - Sequence
4. Debug 输出保留 `DebugLabel` 和失败原因。

验收：

- 当前代码组合可以自然映射成行为树节点。
- 后续可视化只需要替换组合声明，不需要重写攻击 / 移动 / 条件原子。

迁移完成后，`EnemyAutoAISystem.cs` 应只保留：

- 全局配置。

- Tick 调度。
- 与旧系统兼容的外部配置入口。

## 当前风险

1. `SpawnSystem` 生成敌人时 `SourceTagId` 默认写为 `Enemy.Go`，如果 difficulty profile roll 出其他 BP，目前主要靠 BP 名字识别行为。重构时应先收束到 `EnemyProfileResolver`，后续再修为 BP 显式声明 AI profile。
2. `Ertong` 分裂当前直接改世界，和 Intent / Resolution 约定不一致。
3. 如果过早资产化 pipeline，会同时承担代码重构和编辑器设计两种风险。
4. 如果每个行为步骤直接申请 Intent 池对象，未来 dry-run / 可视化预览会比较难做。

## 不做事项

本轮重构不做：

- 行为树编辑器。
- 可视化节点图。
- ScriptableObject 行为树资产。
- 大量新增敌人。
- Boss AI。
- 完整黑板调试 UI。

## 最小完成定义

第一轮完成后，应该达到：

1. `EnemyAutoAISystem` 不再按敌人类型散落判断。
2. 攻击验证、攻击 payload、移动查询、移动 payload 明确分层。
3. 现有 6 类敌人的行为保持一致。
4. 新增一个“同模式但不同参数”的敌人，不需要修改主 AI 调度流程。
5. 当前代码组合可以自然演进到选项式状态机或行为树。
