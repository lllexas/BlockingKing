# 提交前 产品化收束优先级

当前目标：技术策划笔试提交前，把已有系统收束成一个评审可连续体验的产品级 demo。

时间紧时只按优先级执行。P0 未完成时，不做 P2/P3。

## P0：不做会影响评审正常体验

### 1. 达成胜利条件后的 Tick 锁定

问题：

- 胜利条件达成后，如果 tick/input/intent 继续跑，可能出现胜利后敌人继续行动、玩家还能操作、状态被二次修改。

目标：

- 达成胜利条件后锁定 stage tick。
- 禁止玩家继续提交移动/卡牌/noop。
- 停止敌人 AI 继续生成 intent。
- 保留必要的结算表现。

验收：

- 胜利触发后再按 WASD/右键/卡牌不会改变关卡状态。
- 不再产生新 enemy intent。
- 不再推进 spawn counter。
- 能进入胜利/结算界面。

### 2. 死亡界面

问题：

- 玩家/核心箱死亡后需要明确反馈，否则评审会觉得流程断了。

目标：

- 玩家死亡或 core box 归零时进入死亡状态。
- 锁定输入和 tick。
- 显示失败界面。
- 提供重新开始按钮。

验收：

- 死亡后不能继续操作。
- UI 明确显示失败。
- 重新开始可用。

### 3. 开始游戏界面

问题：

- 不能让评审一进场就暴露调试态或不知道如何开始。

目标：

- 有一个正式的开始界面。
- 能开始 run。
- 能进入设置。
- 能退出程序。

验收：

- 打开默认 scene 后，第一眼知道如何开始。
- 不需要 Inspector 操作。
- 不需要看 Console。

### 4. 箱子被推动后获得护盾

问题：

- 这是当前玩法的核心机制之一：推箱子不只是移动空间对象，还应该给箱子/核心带来防御收益。
- 如果缺失，玩家对“为什么要推箱子、如何抵抗敌人”的理解会弱。

目标：

- 箱子被推动后获得护盾。
- 护盾能参与伤害结算。
- UI/数值显示能反映护盾变化。

待确认细节：

- 普通箱子和核心箱子是否都获得护盾。
- 每次推动获得多少护盾。
- 护盾是永久到本关结束，还是每 tick/每回合衰减。
- 多箱连推时每个被推动箱子是否都加盾。

验收：

- 玩家推动箱子后能看到护盾增加。
- 敌人攻击箱子时优先扣护盾。
- 机制对 classic/escort 都生效或明确区分。

### 5. 打包并发给群友试玩

问题：

- 只在 Editor 里跑通不等于产品级。必须有实际 Player 包验证。
- 群友试玩能快速暴露入口、引导、崩溃、画面、音量、难度的真实问题。

目标：

- 完成一次可运行 Windows build。
- 自己本机从 build 启动并跑通至少一轮。
- 发给群友试玩，收集明显 bug 和卡点。

验收：

- build 能启动。
- 默认流程不需要 Unity Editor。
- 关卡内视觉、shader、音频、UI 正常。
- 群友能独立开始游戏并进入战斗。

## P1：强影响产品感和可读性

### 1. UI 补全

范围：

- round 选择界面
- 商店界面
- 不期而遇界面

问题：

- round 选择时需要盖住 stage 场地，避免旧场地和选择 UI 混在一起。
- 商店界面还需要重绘，不能像调试面板。
- 不期而遇界面也需要重绘，保持和 round flow 一致。

目标：

- Round 状态 A/B/C 的 UI 形成明确层级。
- 选择界面遮罩 stage 场地。
- 商店和不期而遇界面具备最低产品可读性。

验收：

- 玩家在 round 选择时不会误以为还能操作 stage。
- 商店购买流程清晰。
- 不期而遇选择/继续流程清晰。

### 2. 玩家与敌人模型补全

对应计划：

优先级：P1

重点：

验收：

- 命中特效和 hit 音效语义对齐。

### 4. 反复游玩优化难度曲线

问题：

- 当前配置系统已经能驱动 round、关卡筛选、刷怪、奖励，但最终体验需要通过反复试玩校准。

目标：

- 明天用默认配置反复跑完整流程。
- 调整 round 难度梯度、spawn timing、奖励、关卡筛选、classic/escort 交替收益。
- 保证前几轮不劝退，中段有压力，后段有高潮。

验收：

- 至少完整跑通多次默认 run。
- 记录明显过难/过易/节奏拖沓点。
- 修改对应 SO 配置，而不是临时改代码。

### 5. 肉鸽难度曲线和道中 round 配置说明书

问题：

- 当前配置系统已经收束，但 round、难度、刷怪、奖励、关卡筛选之间的关系需要说明书，否则后续调参容易迷路。

目标：

- 写一份面向自己/评审/后续维护的配置说明。
- 解释当前 round flow：`A -> B -> A -> B...` 如何推进。
- 解释 classic / escort / skip / shop / event 的配置入口。
- 解释 roundIndex、roundCount、level feature filter、spawn timing、reward pool 如何共同形成难度曲线。

验收：

- 明确写出每个关键 SO 的职责。
- 明确写出“调难度时应该改哪里”。
- 给出一套默认曲线设计意图，例如前期教学、中期压力、后期高潮。
- 不要求长，但要能指导明天/后天继续调参。

## P2：润色与展示增强

### 1. Suno BGM 生成

对应计划：

- `Plan/明日_Suno生成BGM事项.md`

优先级：P2

说明：

- BGM 能提升观感，也能展示 BGM BPM 与 beat time 联动。
- 但系统已接好，不应压过 P0/P1。

### 2. 地图编辑器升级版：3D 渲染状态下编辑

优先级：P2

目标：

- 在接近实际关卡显示的 3D 渲染状态下编辑地图。
- 让编辑器展示更接近最终产品画面。
- 方便边看最终视觉边摆放墙、目标、出生点、特殊单位。

原因：

- 这是很强的展示点，但当前已有 2D/TileMap 编辑器可用。
- 不影响基础提交闭环，因此不是 P0/P1。

验收：

- 能在 3D 视图中看到当前地图渲染。
- 能进行基本编辑或至少快速跳转/同步到编辑状态。
- 不破坏现有编辑器工作流。

### 3. TileMap tilegrid 资源补全与 2D 地图编辑器优化

优先级：P2

目标：

- 补齐 TileMap / TileGrid 相关资源。
- 优化 2D 地图编辑器的可读性和操作效率。
- 让编辑器看起来更像内容生产工具，而不是调试面板。

原因：

- 编辑器已经能用，资源补全属于展示和效率优化。
- 若 P0/P1 未完成，暂不投入太多时间。

验收：

- tile palette / tilegrid 资源更完整。
- 2D 编辑器中地形、单位、目标、出生点更容易辨认。
- 不引入资源引用丢失或 meta 问题。

### 4. SpaceUIAnimator 旋转字段语义整理

优先级：P2

问题：

- `SpaceUIAnimator` 基类有 `_targetRotationY` 字段，但它是运行时旋转目标，不是 prefab 初始姿态。
- 当前如果普通 UI 直接走默认 `FadeIn()`，会按 `_targetRotationY` 写入目标旋转；字段默认 0 时容易把 prefab 上手工摆好的 Rotation Y 洗掉。
- 调整 `_targetRotationY` 在编辑器 Scene 里不可见，容易误以为它是摆版字段。

当前临时处理：

- 主菜单、设置面板、唱片机等不需要运行时旋转的 UI 走保留初始旋转的 FadeIn 路径。
- 暂不大改基类，避免提交前扩大风险。

目标：

- 明确区分“编辑器摆版 Transform Rotation”和“运行时 FadeIn 旋转目标”。
- 将 `_targetRotationY` 的语义改成更清楚的字段，例如 `fadeInTargetRotationY`。
- 增加显式开关，例如 `useFadeInTargetRotation`，默认关闭。
- 需要运行时翻转/转向的 UI 再主动开启。

验收：

- 普通 UIAnimator 不会因为默认 `FadeIn()` 覆盖 prefab 的 Rotation Y。
- 需要旋转动画的 UI 仍可通过明确字段配置。
- Inspector 文案能说明该字段不是编辑器摆版姿态。

## P3：添头，来得及再做

### 1. 藏品 / 遗物实际效果系统

当前状态：

- 商店可以刷新藏品/遗物。
- 可以购买。
- 但缺少实际效果执行系统。

优先级：P3

原因：

- 它是肉鸽长期成长的重要系统，但对当前笔试 demo 不是核心闭环。
- 如果强行做，很容易变成半成品效果表和大量边界 bug。

可接受处理：

- 保留购买和持有展示。
- 明确作为“后续扩展系统”。
- 不在提交前硬塞复杂效果。

如果有余力，最低实现：

- 做 2-3 个非常硬编码、低风险的遗物效果，例如：
  - 初始金币 +N
  - 每次胜利回血 +N
  - 购买卡牌折扣
