# 明日 教程 P0 方案

当前结论：教程是 P0，但不应该立刻开工。必须先把胜利锁定、失败结算、开始界面、回合选择表现、摄像机开局引导、核心机制反馈这些基础体验稳定下来，再做教程。

原因很简单：教程不是给系统打补丁，而是把稳定玩法按顺序交给玩家。如果底层表现还在变化，教程会反复返工，而且会把未稳定的问题包装成“玩家没学会”。

## P0 定位

教程的目标不是讲完整规则，而是让评审能在第一次游玩中理解三件事：

- 自己每拍可以做什么。
- 推箱子为什么重要。
- 敌人、核心箱、胜利/失败条件如何影响决策。

教程完成前，demo 仍然可能可玩；教程完成后，demo 才更像一个能交给陌生人的版本。

## 开工前置条件

以下内容稳定后再进入教程施工：

- 胜利触发后 tick/input/intent 锁定可靠，且有短暂停顿展示胜利。
- 死亡/核心箱归零后有失败界面，且能重新开始。
- 开始界面能进入正式 run，不暴露调试态。
- 回合选择阶段的 UI 背景、按钮、过渡表现稳定。
- 摄像机在 round 选择阶段不漂移，关卡开局能先总览地图再对齐玩家。
- 推箱子获得护盾的表现和数值反馈稳定。
- 敌人意图、远程单位攻击、核心箱受击这些关键反馈不再大改。

## 方案 A：单独的操作教学序章

形式：

- 在正式 run 前放一个固定教学关。
- 地图小、敌人少、步骤可控。
- 只教学基础操作、推箱子、敌人回合、核心保护、胜利目标。
- 完成后进入正式 run 的第一轮选择。

优点：

- 最可控，最适合评审第一次打开 demo。
- 可以强制按顺序展示机制，不受随机 run 干扰。
- 教学失败时可以快速重开，不污染正式流程。

缺点：

- 需要额外做一套入口和完成后的流程跳转。
- 如果教学关太长，会拖慢正式体验开始。
- 单独序章如果和正式关卡差异太大，会产生割裂感。

适用判断：

- 如果提交版本只允许评审玩一次，优先选这个。
- 如果核心操作还需要明确手把手说明，优先选这个。

## 方案 B：把教学放进正式流程中

形式：

- 正式 run 第一关就是低压教学关。
- 用 UI 提示、阶段性目标、敌人配置和地图布局逐步教学。
- 教学结束后不切场，直接继续 run 流程。

优点：

- 不打断正式体验，玩家从第一秒就在玩真正的流程。
- 不需要维护“教学模式”和“正式模式”两套感觉。
- 更适合肉鸽结构，教学内容可以自然嵌入第一轮。

缺点：

- 对流程控制要求更高。
- 如果随机内容介入太早，玩家可能没学会核心规则就被干扰。
- 教学提示需要能和正式 UI 共存，不能挡住操作。

适用判断：

- 如果 run 流程和第一关已经足够稳定，可以选这个。
- 如果想突出“这是完整游戏的一部分”，选这个。

## 当前推荐

提交前版本优先采用方案 A：单独的操作教学序章。

理由：

- 当前目标是让陌生评审稳定理解核心体验。
- 序章更容易控制变量，能避开随机流程、难度曲线和奖励选择的不确定性。
- 序章结束后再进入正式 round 选择，可以把教程和主循环清楚衔接起来。

但实现时不要做成完全独立的一套系统。推荐复用正式 LevelPlayer、DrawSystem、CameraController、RunRoundController 的已有能力，只在流程入口和教学提示上加薄层。

## 最小教学内容

第一段：移动与节拍

- 玩家看到自己、核心箱、目标点或出口。
- 提示每次操作推进一拍。
- 让玩家移动一次，观察回合推进。

第二段：推箱子

- 放一个必须推动的箱子。
- 推动后展示箱子位置变化。
- 如果护盾机制已完成，同步展示护盾增加。

第三段：敌人意图

- 放一个低威胁近战敌人。
- 让玩家看到敌人意图。
- 玩家通过移动或推箱子处理威胁。

第四段：保护核心

- 让敌人接近核心箱，但不要立刻造成失败。
- 强调核心箱受击/护盾/血量反馈。

第五段：胜利

- 完成目标后触发胜利锁定。
- 保留 4 beat time 的胜利展示停顿。
- 进入正式 run round 选择。

## 实施拆分

### 1. 教学入口

目标：

- 开始游戏后先进入教学序章。
- 教学完成后进入正式 run。
- 之后可以考虑加“跳过教程”，但提交前不是必须。

验收：

- 默认新开局会先进入教学。
- 教学胜利后能进入正式 round 选择。
- 教学失败后能重试教学。

### 2. 教学关卡数据

目标：

- 做一张固定小地图。
- 只放最少必要对象。
- 保证摄像机 overview 能完整展示。

验收：

- 不依赖随机生成。
- 开局镜头总览清楚。
- 玩家不会被第一步误导。

### 3. 教学提示层

目标：

- 用正式 UI 风格做少量提示。
- 提示由教学状态驱动，不写死在通用战斗 UI 内。
- 提示不遮挡关键棋盘格和手牌。

验收：

- 每个教学阶段只有一个明确提示。
- 完成条件达成后提示能自动进入下一步。
- 战斗锁定/胜利锁定期间不接受额外教学输入。

### 4. 教学状态机

目标：

- 记录当前教学阶段。
- 监听玩家移动、推箱子、敌人意图、核心受击、胜利结算等事件。
- 控制提示推进和必要的软限制。

验收：

- 不靠帧轮询猜状态。
- 不破坏正式战斗逻辑。
- 教学完成后能干净退出。

## 暂不做

- 不做长篇文本剧情。
- 不做复杂分支教学。
- 不做完整百科或规则书。
- 不做每个敌人类型的逐一教学。
- 不做强制重写现有战斗系统的教程专用版本。

## 风险

- 如果推箱子护盾、敌人意图、失败界面还没稳定，教程文本会很快过期。
- 如果摄像机开局引导不稳定，教学第一眼会失败。
- 如果教学提示和回合选择 UI 走两套风格，demo 会显得割裂。

## 暂定验收

- 一个没看过项目的人打开 demo 后，不需要解释即可完成教学序章。
- 教学结束后自然进入正式 run。
- 教学过程中不会出现胜利后还能操作、失败后流程卡死、镜头漂移、提示挡住关键操作等问题。
- 教学总时长控制在 1 到 2 分钟内。

## 当前分镜草案

### 第一幕

“推过去”

目标：

- 只教最基础的移动与推箱。
- 玩家做一次最直接的推动作。
- 不引入卡牌、不引入敌人、不引入结算按钮。

### 第二幕

“这是个死局，在其他的推箱子游戏中，我会建议你重新开始。但在《推王之王》中，你不需要被这些繁文缛节约束。把墙撞开！！！”

目标：

- 教玩家本作不是纯传统 Sokoban。
- 明确卡牌/撞墙/破坏规则才是《推王之王》的核心乐趣之一。
- 这一幕适合发一张能撞墙或破墙的卡，并强制玩家用一次。

### 第三幕

“这是经典关卡，我暂时收走了你的卡牌，请尝试着把所有箱子推到目标点。如果你发现关卡没法再进行，就点击结算按钮吧。量力而行，不要内耗。”

目标：

- 教经典关卡的第一阶段。
- 教玩家卡牌并非全程可用。
- 教玩家在经典模式里，第一阶段结束依赖“结算”按钮，而不是无限硬解。

### 第四幕

“现在，敌人会周期性生成在目标点。卡牌系统已回归，请消灭所有敌人，并把所有箱子推到目标点。”

目标：

- 教经典关卡的第二阶段。
- 教玩家敌人会生成、卡牌回归、胜利条件升级。
- 明确不是只杀敌，也不是只推箱，而是两个条件都要完成。

### 第五幕

“不知道你发现没有，战车卡可以一次把箱子推好远。主教卡也是。你可以试试看。”

目标：

- 教玩家开始理解不同棋子卡的空间杠杆。
- 重点展示战车卡和主教卡在推箱上的效率。
- 这一幕更像引导式实验，不一定需要失败压力。

### 第六幕

“这是押送关卡，你需要把蓝色的核心箱推到蓝色的核心目标点。请善用战车与主教卡，不然敌人会把你淹没。”

目标：

- 教押送关卡的核心目标。
- 区分普通箱子与核心箱。
- 强调押送模式的节奏压力来自敌人持续涌入，因此需要高效率卡牌。

### 结尾

教程结束。

当前理解：

- 前四幕构成 classic 教学主线。
- 第五幕是卡牌效率认知补充。
- 第六幕单独切到 escort 教学。
- 整体已经足够作为 tutorial director 的第一版分镜输入。
